# Takeaways

## TLDR

### 1 Task = 1 Core = 1 Thread - process 1 Partition

This is the fundamental unit of parallelism in Spark. Each task runs on a single thread, processes one partition, and occupies one core (controlled by `spark.task.cpus`, default `1`).

```text
Executor 1 (2 cores)          Executor 2 (2 cores)
┌──────────┬──────────┐      ┌──────────┬──────────┐
│ Core 1   │ Core 2   │      │ Core 1   │ Core 2   │
│ Thread 1 │ Thread 2 │      │ Thread 3 │ Thread 4 │
│ Task A   │ Task B   │      │ Task C   │ Task D   │
│ Part 0   │ Part 1   │      │ Part 2   │ Part 3   │
└──────────┴──────────┘      └──────────┴──────────┘
        4 partitions processed simultaneously
```

If there are more partitions than cores (e.g., 20 partitions with 4 cores), they queue up and run in waves of 4.

### Computer Architecture Basics (0.prerequisite.md)

- **Core** = independent processing unit in a CPU. More cores = more things at once.
- **Process** = a running program with its own isolated memory (e.g., a Spark executor is a JVM process).
- **Thread** = lightweight unit of work inside a process. Threads share the process's RAM.
- **Multithreading** = multiple threads running in one process on different cores. This is how executors run tasks.
- **JVM** = Java Virtual Machine, the process that runs Spark code. Even PySpark sends instructions to a JVM via Py4J.
- **RAM** = fast, volatile memory for active data (~100 ns access). **Disk** = slow, persistent storage (~0.1 ms SSD).
- **OOM** = Out Of Memory. Spark spills to disk first; if that's not enough, the executor crashes.

### Cluster Architecture (2.cluster.md)

- The **Docker Compose** defines a **machine cluster** (containers with CPU/RAM). This is NOT the Spark cluster.
- The **Spark cluster** is JVM processes (driver + executors) that run on top of those machines.
- The **cluster manager** (standalone master at `:7077`) brokers resources between them.
- **Ports**: `:8080` Master Web UI, `:7077` Master RPC, `:18080` History Server.
- **History Server** = read-only UI that persists after jobs finish. Reads event logs from shared `spark-events` volume.

### Resource Request (4.resource-request.md)

- Machines **offer** resources (workers: 4 cores, 4g each). Spark app **requests** a subset.
- `spark.executor.instances=2` → 2 executor JVMs across the cluster.
- `spark.executor.cores=2` → 2 cores per executor → 2 tasks in parallel per executor.
- `spark.executor.memory="2g"` → JVM heap per executor. Usable = `(2048-300) × 0.6 = 1048 MB`.
- `spark.cores.max=4` → caps total cores (standalone Spark is greedy by default).
- `spark.sql.files.maxPartitionBytes=256MB` → partition size → determines number of tasks.
- `spark.task.cpus=1` (default, not in code) → cores per task. Tasks per executor = `executor.cores ÷ task.cpus`.

### Running the Application (5.running-application.md)

- **Parquet on disk** is compressed (SNAPPY, dictionary encoding). **In-memory** DataFrame is ~8-10x larger.
- **Caching** (`df.cache()` + `df.count()`) materializes data in RAM. Visible in History Server Storage tab.
- If cached data exceeds executor memory, partitions spill to disk.

### Partitions (6.partitions.md)

- A **partition** = a chunk of data processed by 1 task.
- Execution hierarchy: **Application → Job → Stage → Task**.
- **Shuffle boundaries** (groupBy, join, repartition) create new stages.
- Larger partitions = fewer tasks but more memory per task and higher spill risk.
- Smaller partitions = more tasks but lighter workload per task.

### Execution Plan (7.execution-plan.md)

- Spark uses **two-phase aggregation**: partial aggregate (map-side) → shuffle → final aggregate (reduce-side).
- **HashAggregate** = fast (in-memory hash map), used for numeric/fixed-size types.
- **SortAggregate** = slower (requires sorting), used for variable-size types like Strings.
- **Exchange (shuffle)** = redistribute data by key across executors. Controlled by `spark.sql.shuffle.partitions` (default 200).
- **AQE** (Adaptive Query Execution) = runtime optimization that coalesces small shuffle partitions.

## 1. Spark Job Structure

A Spark job is divided into multiple **stages**, each containing multiple **tasks**.

```text
Job
 ├── Stage 0
 │    ├── Task 0  →  [Partition 0]
 │    ├── Task 1  →  [Partition 1]
 │    └── Task 2  →  [Partition 2]
 └── Stage 1
      ├── Task 0  →  [Partition 0]
      └── Task 1  →  [Partition 1]
```

- A **task** is the smallest unit of work in Spark.
- Each task processes exactly one **partition** (a piece of data).
- Tasks within a stage can run **in parallel** across executors.

## 2. Executor Parallelism via Multithreading

Tasks are handled in parallel inside an executor using the **multithreading** paradigm.

```text
Executor (JVM Process)
 ├── Thread 1  →  Task A
 ├── Thread 2  →  Task B
 └── Thread 3  →  Task C
        ↑
   Each thread runs one task concurrently
```

## 3. Executor Cores vs Memory Trade-off

Increasing the number of executor cores increases parallelism, since each task runs on a defined number of cores (**1 by default**, controlled by `spark.task.cpus`).

**However**, increasing cores while keeping memory the same **shrinks each task's memory share**, because more tasks now compete for the same memory pool.

### 3.1 Example: 2 cores, 2g memory

```text
Executor [2 cores, 2g memory]
 ┌─────────────────┬─────────────────┐
 │   Task 1        │   Task 2        │
 │   1 core        │   1 core        │
 │   ~1g memory    │   ~1g memory    │
 └─────────────────┴─────────────────┘
```

### 3.2 Example: 4 cores, 2g memory (same memory, more cores)

```text
Executor [4 cores, 2g memory]
 ┌────────┬────────┬────────┬────────┐
 │ Task 1 │ Task 2 │ Task 3 │ Task 4 │
 │ 1 core │ 1 core │ 1 core │ 1 core │
 │ ~512mb │ ~512mb │ ~512mb │ ~512mb │
 └────────┴────────┴────────┴────────┘
   -> Each task gets less memory!
```

> **Key insight**: More cores = more parallelism, but each task gets a smaller share of the available memory. If tasks are memory-intensive (such as those involving large joins or shuffles), they may consume more memory than what is allocated to them. When this happens, Spark first tries to spill extra data to disk. If there still isn't enough memory, the process can run out of memory (OOM), causing the executor to crash and the job to fail with an "Out Of Memory" error.

## 4. Executor Memory Breakdown

Increasing executor memory gives each task a larger memory portion, reducing data spills and improving performance.

The memory available for task processing and storage is **not** the full `spark.executor.memory`. It's calculated as:

```text
   Usable Memory = (spark.executor.memory - Reserved Memory) × spark.memory.fraction

   Defaults:
   Reserved Memory  = 300 MB
   Memory Fraction  = 0.6
```

### 4.1 Example: 2g executor memory

```text
spark.executor.memory = 2g (2048 MB)

┌──────────────────────────────────────────────────┐
│              Executor JVM (2048 MB)              │
│                                                  │
│  ┌─────────────────────────────────────────────┐ │
│  │        Reserved Memory (300 MB)             │ │
│  └─────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────┐ │
│  │   Usable: (2048 - 300) × 0.6 = 1048.8 MB    │ │
│  │  ┌───────────────────┬───────────────────┐  │ │
│  │  │ Storage Memory    │ Execution Memory  │  │ │
│  │  │ (caching, etc.)   │ (shuffles, joins) │  │ │
│  │  └───────────────────┴───────────────────┘  │ │
│  └─────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────┐ │
│  │   User Memory: (2048 - 300) × 0.4 = 699 MB  │ │
│  │   (internal metadata, UDFs, etc.)           │ │
│  └─────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────┘
```

## 5. Partition Size and Task Workload

The partition size determines the workload for a single task and the total number of tasks.

```text
Larger partitions                    Smaller partitions
┌──────────────────────┐            ┌──────┬──────┬──────┬──────┐
│     Partition 1      │            │  P1  │  P2  │  P3  │  P4  │
│     (256 MB)         │            │ 64MB │ 64MB │ 64MB │ 64MB │
├──────────────────────┤            ├──────┼──────┼──────┼──────┤
│     Partition 2      │            │  P5  │  P6  │  P7  │  P8  │
│     (256 MB)         │            │ 64MB │ 64MB │ 64MB │ 64MB │
└──────────────────────┘            └──────┴──────┴──────┴──────┘
  2 tasks total                       8 tasks total
  ⚠ Longer per-task time             ✓ Shorter per-task time
  ⚠ Higher spill risk                ✓ Lower spill risk
  ✓ Less scheduling overhead         ⚠ More scheduling overhead
```

- **Larger partitions** → fewer tasks, but each task takes longer and is more likely to spill data to disk.
- **Smaller partitions** → more tasks, but each task has a lighter workload and lower chance of spilling.

> Controlled by `spark.sql.files.maxPartitionBytes` (default 128 MB).

## 6. Data Types Affect Aggregation Strategy

The data type of the column being aggregated affects which aggregation algorithm Spark uses.

```text
Numeric columns (int, double, etc.)        String columns
┌─────────────────────────┐                ┌─────────────────────────┐
│     HashAggregate       │                │     SortAggregate       │
│                         │                │                         │
│  ┌───────────────────┐  │                │  1. Sort all rows       │
│  │   Hash Table      │  │                │  2. Walk sorted data    │
│  │  key → accumulator│  │                │  3. Aggregate groups    │
│  │  "A" → sum=150    │  │                │                         │
│  │  "B" → sum=230    │  │                │  ⚠ Requires full sort   │
│  └───────────────────┘  │                │  ⚠ Slower than hashing  │
│                         │                │                         │
│  ✓ Fast (O(1) lookups)  │                │  ⚠ Slower (O(n log n))  │
│  ✓ No sorting needed    │                │                         │
└─────────────────────────┘                └─────────────────────────┘
```

- **HashAggregate**: Used for numeric types. Maintains an in-memory hash map for fast lookups — very efficient.
- **SortAggregate**: Used for complex types like strings. Requires sorting all data first, making it significantly slower.

> **Tip**: If possible, avoid aggregating on string columns. Consider encoding strings as numeric IDs before aggregation.
