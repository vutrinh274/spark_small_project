# Partitions and Spark Architecture

## 1. What is a Partition?

A partition is a chunk of data assigned to a single task. Spark splits input data into partitions so multiple tasks can process them in parallel.

```text
10.4 GB of Parquet data (4 files)
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Split into partitions (maxPartitionBytes = 256 MB):         │
│                                                              │
│  ┌───────┬───────┬───────┬───────┬───────┬─────┬───────┐     │
│  │Part 0 │Part 1 │Part 2 │Part 3 │Part 4 │ ... │Part 41│     │
│  │256 MB │256 MB │256 MB │256 MB │256 MB │     │~256 MB│     │
│  └───────┴───────┴───────┴───────┴───────┴─────┴───────┘     │
│                                                              │
│  ~10.4 GB ÷ 256 MB ≈ 42 partitions = 42 tasks                │
└──────────────────────────────────────────────────────────────┘
```

The key rule: **1 partition = 1 task**. The number of partitions determines how many tasks Spark creates for a stage.

## 2. How Partition Size is Controlled

The config `spark.sql.files.maxPartitionBytes` controls the maximum size of each partition when reading files. We set it to 256 MB in `main.py`:

```python
SPARK_SQL_FILES_MAXPARTITIONBYTES = (
    256 * 1024 * 1024
)  # 256 MB
```

### 2.1. Smaller partitions vs Larger partitions

```text
Smaller partitions (e.g., 64 MB):
┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬─── ...
│ P0 │ P1 │ P2 │ P3 │ P4 │ P5 │ P6 │ P7 │ P8 │ P9 │
│64MB│64MB│64MB│64MB│64MB│64MB│64MB│64MB│64MB│64MB│    ~163 partitions
└────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴───
  ✓ Each task uses less memory
  ✓ Better load balancing (more granular)
  ✗ More scheduling overhead (163 tasks to manage)
  ✗ More task launch latency

Larger partitions (e.g., 512 MB):
┌──────────┬──────────┬──────────┬──────────┬──── ...
│   P0     │   P1     │   P2     │   P3     │
│  512 MB  │  512 MB  │  512 MB  │  512 MB  │     ~21 partitions
└──────────┴──────────┴──────────┴──────────┴────
  ✓ Less scheduling overhead (fewer tasks)
  ✓ Less task launch latency
  ✗ Each task needs more memory → higher OOM risk
  ✗ Worse load balancing (fewer, chunkier tasks)
```

### 2.2. Partition size vs per-task memory

The partition size must fit within the per-task memory budget. From [doc 0](./0.prerequisite.md), each task gets roughly:

```text
Per-task memory ≈ Unified Memory ÷ number of cores per executor

With our config (2 GB executor, 2 cores):
  Unified Memory = (2048 - 300) × 0.6 = ~1049 MB
  Per-task memory = ~1049 ÷ 2 = ~524 MB

256 MB partition → fits in 524 MB ✓
512 MB partition → tight fit, risky on shuffle/join ⚠
1 GB partition   → does not fit → OOM ✗
```

> Partition size should leave headroom for intermediate data (shuffle buffers, hash tables, sort arrays) — not just the raw partition data.

## 3. Spark Architecture Review

Before looking at how partitions flow through Spark, let's understand the execution hierarchy:

```text
Application
└── Job
    └── Stage
        └── Task
```

### 3.1. Application

An application is the entire program submitted via `spark-submit`. It creates a SparkSession, requests cluster resources (executors), and runs one or more jobs.

```text
spark-submit main.py
      │
      ▼
┌─────────────────────────────────────┐
│  Application: "my_app"              │
│                                     │
│  Driver (on spark-master)           │
│  Executor 1 (on worker-1)          │
│  Executor 2 (on worker-2)          │
│                                     │
│  Runs until spark.stop() is called  │
└─────────────────────────────────────┘
```

### 3.2. Job

A job is triggered by an **action** — an operation that returns a result or writes output. Every action creates a new job.

```text
Common actions that trigger a job:
  .count()    → returns a number
  .show()     → prints rows to console
  .collect()  → returns all rows to driver
  .write()    → writes data to disk

Transformations do NOT trigger a job (they are lazy):
  .groupBy()  → lazy, just records the plan
  .filter()   → lazy
  .select()   → lazy
  .agg()      → lazy
```

In our `main.py`:

```text
df = spark.read.parquet(...)   ← transformation (lazy)
df.cache()                     ← transformation (lazy, marks for caching)
df.count()                     ← ACTION → triggers Job 0

result = df.groupBy(...).agg(...)  ← transformation (lazy)
result.show(10)                    ← ACTION → triggers Job 1
```

```text
Application "my_app"
├── Job 0: triggered by count()
│   └── Read parquet → cache → count
│
└── Job 1: triggered by show()
    └── Read cached data → groupBy → agg → show
```

### 3.3. Stage

A job is divided into **stages** at **shuffle boundaries**. A shuffle happens when data needs to be redistributed across partitions (e.g., `groupBy`, `join`, `repartition`).

```text
Job 1: df.groupBy("l_linenumber").agg(...).show()

Stage 0 (before shuffle):              Stage 1 (after shuffle):
┌─────────────────────────┐            ┌─────────────────────────┐
│  Read partitions        │            │  Receive shuffled data  │
│  Apply local operations │  shuffle   │  Perform aggregation    │
│  (filter, map, etc.)    │ ────────►  │  Return results         │
│                         │            │                         │
│  Each task reads its    │            │  Each task handles one  │
│  own partition          │            │  group of keys          │
└─────────────────────────┘            └─────────────────────────┘
```

Why does `groupBy` cause a shuffle?

```text
Before shuffle (data is spread randomly across partitions):

Partition 0: [line=1, line=3, line=1, line=2]
Partition 1: [line=2, line=1, line=3, line=3]
Partition 2: [line=1, line=2, line=2, line=1]

    │  groupBy("l_linenumber") → SHUFFLE
    │  All rows with the same key must go to the same partition
    ▼

After shuffle (data is grouped by key):

Partition 0: [line=1, line=1, line=1, line=1, line=1]  ← all line=1
Partition 1: [line=2, line=2, line=2, line=2]           ← all line=2
Partition 2: [line=3, line=3, line=3, line=3]           ← all line=3

Now each task can compute the aggregation for its key group.
```

### 3.4. Task

A task is the smallest unit of execution. Each task processes **one partition** in **one stage**.

```text
Stage 0: Read parquet (42 partitions → 42 tasks)

With 4 total cores (2 executors × 2 cores):

Wave 1:  Task 0, Task 1, Task 2, Task 3     ← 4 tasks run in parallel
Wave 2:  Task 4, Task 5, Task 6, Task 7
Wave 3:  Task 8, Task 9, Task 10, Task 11
  ...
Wave 11: Task 40, Task 41                   ← last wave (only 2 tasks)

42 tasks ÷ 4 cores = 11 waves (10 full + 1 partial)
```

```text
Executor 1 (2 cores)                 Executor 2 (2 cores)
┌───────────┬───────────┐           ┌───────────┬───────────┐
│  Core 0   │  Core 1   │           │  Core 0   │  Core 1   │
│           │           │           │           │           │
│  Task 0   │  Task 1   │           │  Task 2   │  Task 3   │
│  Part 0   │  Part 1   │           │  Part 2   │  Part 3   │
│  256 MB   │  256 MB   │           │  256 MB   │  256 MB   │
└───────────┴───────────┘           └───────────┴───────────┘
     Sharing 2 GB RAM                    Sharing 2 GB RAM
```

## 4. Putting It All Together

Here's how our application flows from submission to completion:

```text
1. spark-submit main.py
   → Application "my_app" starts
   → Driver requests 2 executors (2 cores, 2 GB each)

2. df.count() triggers Job 0:
   ┌─────────────────────────────────────────────────┐
   │  Job 0                                          │
   │  ┌───────────────────────────────────────────┐  │
   │  │  Stage 0: Read parquet + cache + count    │  │
   │  │                                           │  │
   │  │  10.4 GB parquet → 42 partitions          │  │
   │  │  42 tasks, 4 cores → ~11 waves            │  │
   │  │  Each task: read 256 MB, cache, count     │  │
   │  └───────────────────────────────────────────┘  │
   └─────────────────────────────────────────────────┘

3. result.show() triggers Job 1:
   ┌─────────────────────────────────────────────────┐
   │  Job 1                                          │
   │  ┌───────────────────────────────────────────┐  │
   │  │  Stage 1: Read from cache                 │  │
   │  │  42 tasks (1 per cached partition)        │  │
   │  │  Each task: local groupBy pre-aggregation │  │
   │  └──────────────────┬────────────────────────┘  │
   │                     │ SHUFFLE                   │
   │  ┌──────────────────▼────────────────────────┐  │
   │  │  Stage 2: Aggregate + show                │  │
   │  │  200 tasks (default shuffle partitions)   │  │
   │  │  Each task: final aggregation per key     │  │
   │  └───────────────────────────────────────────┘  │
   └─────────────────────────────────────────────────┘
```

### 4.1. Summary table

| Level | What it is | How many | Determined by |
| --- | --- | --- | --- |
| **Application** | The whole program | 1 per `spark-submit` | You submit it |
| **Job** | Triggered by an action | 1 per action (`count`, `show`, `write`) | Number of actions in your code |
| **Stage** | Split at shuffle boundaries | 1+ per job | Number of shuffles (`groupBy`, `join`) |
| **Task** | Processes one partition | 1 per partition per stage | Number of partitions |
