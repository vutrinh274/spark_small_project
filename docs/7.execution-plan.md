# Execution Plan

## 1. What is an Execution Plan?

An execution plan is the step-by-step strategy Spark creates to execute your DataFrame operations. It shows exactly what Spark will do — and in what order — to produce your result.

To view the physical execution plan, go to the Spark History Server (localhost:18080), choose your application, switch to the "SQL/DataFrame" tab, select the completed query, and click "Details".

---

## 2. Reading the Plan

The plan reads **bottom-up** — the first operation is at the bottom, and the final result is at the top.

```text
== Physical Plan ==
AdaptiveSparkPlan (18)
+- == Final Plan ==
   ResultQueryStage (11), Statistics(sizeInBytes=8.0 EiB)
   +- CollectLimit (10)
      +- SortAggregate (9)
         +- * Sort (8)
            +- AQEShuffleRead (7)
               +- ShuffleQueryStage (6), Statistics(sizeInBytes=148.9 KiB, rowCount=588)
                  +- Exchange (5)
                     +- SortAggregate (4)
                        +- * Sort (3)
                           +- * ColumnarToRow (2)
                              +- Scan parquet (1)
```

This corresponds to our code:

```python
result = df.groupBy("l_linenumber").agg(
    F.sum("l_quantity").alias("sum_l_quantity"),
    F.avg("l_extendedprice").alias("avg_price"),
    F.sum("l_discount").alias("sum_l_discount"),
    ...
)
result.show(10)
```

## 3. Node-by-Node Breakdown

### 3.1. Scan parquet (1)

```text
+- Scan parquet (1)

What: Read parquet files from disk
How:  Scan all files matching the input path
      Only reads the columns needed for the query (column pruning)
```

This is where Spark reads the raw parquet files. Parquet's columnar format allows Spark to skip columns not used in the query — it only reads `l_linenumber`, `l_quantity`, `l_extendedprice`, `l_discount`, and the other columns referenced in the `agg()`.

---

### 3.2. ColumnarToRow (2)

```text
+- * ColumnarToRow (2)

What: Convert data from columnar format to row format
Why:  Parquet stores data column-by-column, but Spark processes data row-by-row
```

```text
Columnar (Parquet on disk):          Row-based (Spark in memory):
┌────────────┬────────────┐         ┌──────────┬──────────┬──────────┐
│ l_orderkey │ l_quantity │         │ Row 0: 1 │    17    │  0.04    │
│     1      │    17      │  ────►  │ Row 1: 1 │    36    │  0.09    │
│     1      │    36      │         │ Row 2: 1 │     8    │  0.10    │
│     1      │     8      │         └──────────┴──────────┴──────────┘
└────────────┴────────────┘
  Columns stored together              Rows stored together
```

> The `*` prefix (e.g., `* ColumnarToRow`) means this node runs in **whole-stage code generation** — Spark compiles multiple operations into a single optimized Java function instead of calling them one-by-one. This is a performance optimization.

---

### 3.3. Sort (3) + SortAggregate (4) — Partial Aggregation

```text
+- SortAggregate (4)
   +- * Sort (3)

What: Sort data by grouping key, then perform PARTIAL aggregation
Why:  Reduces data volume BEFORE the shuffle
```

This is the **Map-side (partial) aggregation**. Before sending data across the network, Spark pre-aggregates locally on each partition:

```text
Partition 0 (before partial agg):
┌────────────────┬────────────┬───────────┐
│ l_linenumber   │ l_quantity │ l_discount│
│      1         │     17     │   0.04    │
│      3         │     36     │   0.09    │
│      1         │      8     │   0.10    │
│      1         │     22     │   0.05    │
│      3         │     14     │   0.02    │
└────────────────┴────────────┴───────────┘
  5 rows

         │  Sort by l_linenumber
         │  Then aggregate per group
         ▼

Partition 0 (after partial agg):
┌────────────────┬──────────────────┬────────────────┐
│ l_linenumber   │ sum(l_quantity)  │ sum(l_discount)│
│      1         │     47           │   0.19         │  ← 3 rows → 1
│      3         │     50           │   0.11         │  ← 2 rows → 1
└────────────────┴──────────────────┴────────────────┘
  2 rows (much less data to shuffle!)
```

Why **Sort** before **SortAggregate**? SortAggregate works by scanning sorted data — when it encounters a new key, it knows the previous group is complete and outputs the partial result. This is memory-efficient because it doesn't need a hash table.

```text
Sorted data:     [1, 1, 1, 3, 3]
                   ^^^^^^^^  ^^^^
                   group 1   group 2

SortAggregate scans left-to-right:
  key=1, key=1, key=1 → output partial result for key=1
  key=3, key=3         → output partial result for key=3
```

---

### 3.4. Exchange (5) — The Shuffle

```text
+- Exchange (5)

What: Redistribute data across the cluster by the grouping key (hash partitioning)
Why:  All rows with the same key must end up on the same partition for final aggregation
```

```text
Before Exchange:                          After Exchange:

Executor 1:                               Executor 1:
  Part 0: [line=1: sum=47, line=3: sum=50]   Part 0: [line=1: sum=47, line=1: sum=39, ...]
  Part 1: [line=1: sum=39, line=2: sum=28]          ← all line=1 rows from all partitions

Executor 2:                               Executor 2:
  Part 2: [line=2: sum=33, line=3: sum=41]   Part 1: [line=2: sum=28, line=2: sum=33, ...]
  Part 3: [line=1: sum=45, line=2: sum=22]          ← all line=2 rows from all partitions

      Data is hash-partitioned by l_linenumber
      hash(key) % num_partitions → determines destination
```

By default, Spark creates **200 shuffle partitions** (`spark.sql.shuffle.partitions = 200`). So the data is hash-distributed into 200 partitions.

> This is the most expensive step — data is serialized, written to disk, transferred over the network, and deserialized on the receiving end.

---

### 3.5. ShuffleQueryStage (6) + AQEShuffleRead (7) — AQE Optimization

```text
+- AQEShuffleRead (7)
   +- ShuffleQueryStage (6), Statistics(sizeInBytes=148.9 KiB, rowCount=588)

What: Adaptive Query Execution reads the shuffle output and optimizes it
Why:  The default 200 shuffle partitions is wasteful for small data
```

This is where **Adaptive Query Execution (AQE)** kicks in. AQE was introduced in Spark 3.0 — it collects **runtime statistics** during execution and adjusts the plan on the fly.

Here's what happens:

```text
Without AQE:
  Exchange produces 200 shuffle partitions
  Most are nearly empty (only 588 rows total!)
  200 tasks created, most do almost nothing

  ┌────┬────┬────┬────┬────┬────┬─── ... ───┬────┐
  │ P0 │ P1 │ P2 │ P3 │ P4 │ P5 │           │P199│  200 partitions
  │data│data│    │    │data│    │           │    │  Most empty!
  └────┴────┴────┴────┴────┴────┴─── ... ───┴────┘

With AQE:
  AQE sees the shuffle output is only 148.9 KiB (588 rows)
  Coalesces 200 partitions into far fewer (possibly 1)

  ┌──────────────────────────────────────────────┐
  │  P0 (coalesced): all 588 rows, 148.9 KiB     │  1 partition
  └──────────────────────────────────────────────┘
  Much more efficient — 1 task instead of 200
```

The `Statistics(sizeInBytes=148.9 KiB, rowCount=588)` tells us the shuffle output is tiny — only 588 rows and 148.9 KiB. This makes sense because the partial aggregation (step 3.3) already reduced millions of rows down to a few hundred partial results (one per `l_linenumber` per partition).

### 3.6. Sort (8) + SortAggregate (9) — Final Aggregation

```text
+- SortAggregate (9)
   +- * Sort (8)

What: Sort by grouping key, then perform FINAL aggregation
Why:  Combine all partial results into the final answer
```

This is the **Reduce-side (final) aggregation**. It merges the partial aggregates from all partitions:

```text
Input (partial results from all partitions):
┌────────────────┬──────────────────┬────────────────┐
│ l_linenumber   │ sum(l_quantity)  │ sum(l_discount)│
│      1         │     47           │   0.19         │  ← from partition 0
│      1         │     39           │   0.15         │  ← from partition 1
│      1         │     45           │   0.21         │  ← from partition 3
│      2         │     28           │   0.12         │  ← from partition 1
│      2         │     33           │   0.14         │  ← from partition 2
│      ...       │     ...          │   ...          │
└────────────────┴──────────────────┴────────────────┘

         │  Sort by l_linenumber
         │  Then merge partials per group
         ▼

Final result:
┌────────────────┬──────────────────┬────────────────┐
│ l_linenumber   │ sum(l_quantity)  │ sum(l_discount)│
│      1         │    131           │   0.55         │  ← 47+39+45
│      2         │     61           │   0.26         │  ← 28+33
│      ...       │     ...          │   ...          │
└────────────────┴──────────────────┴────────────────┘
```

### 3.7. Two-phase aggregation: Why partial + final?

```text
Without partial aggregation:
  Shuffle ALL raw rows across the network
  10.4 GB of data moving between executors
  Slow, expensive, high risk of OOM

  ┌──────────────────┐     SHUFFLE      ┌──────────────────┐
  │ Millions of rows │ ──────────────►  │ Final aggregate  │
  │ (10.4 GB)        │  network traffic │ on one side      │
  └──────────────────┘                  └──────────────────┘

With partial aggregation (what Spark does):
  Pre-aggregate locally first, THEN shuffle the small result
  Only 148.9 KiB moves across the network

  ┌──────────────────┐    ┌─────────────┐    SHUFFLE     ┌──────────────┐
  │ Millions of rows │ ►  │ 588 partial │ ────────────►  │ Final merge  │
  │ (10.4 GB)        │    │ results     │  148.9 KiB     │ (tiny)       │
  └──────────────────┘    └─────────────┘                └──────────────┘
```

### 3.8. SortAggregate vs HashAggregate

Spark provides two approaches to perform aggregation: **SortAggregate** and **HashAggregate**.

#### 3.8.1 SortAggregate

Sorts the data by the grouping key first, then iterates over it. When the key changes, the previous group is complete.

```text
Input (unsorted):  [3, 1, 1, 3, 2, 1]
                         │
                   Sort by key
                         │
                         ▼
Sorted:            [1, 1, 1, 2, 3, 3]
                    ^^^^^^^^  ^  ^^^^
                    group 1  g2  group 3

Scan left-to-right:
  key=1 → accumulate sum
  key=1 → accumulate sum
  key=1 → accumulate sum → key changes → output group 1 result
  key=2 → accumulate sum → key changes → output group 2 result
  key=3 → accumulate sum
  key=3 → accumulate sum → end of data → output group 3 result
```

- Requires sorting the data first (expensive)
- Memory-efficient: only holds one group in memory at a time

#### 3.8.2 HashAggregate

Builds a hash table where each key maps to an aggregation buffer that gets updated in-place.

```text
Input: [3, 1, 1, 3, 2, 1]

Hash table (built incrementally):
┌─────┬──────────────────────────┐
│ Key │ Aggregation Buffer       │
├─────┼──────────────────────────┤
│  3  │ sum=50, count=2          │  ← updated in-place
│  1  │ sum=47, count=3          │  ← updated in-place
│  2  │ sum=28, count=1          │  ← updated in-place
└─────┴──────────────────────────┘

No sorting needed — just hash the key and update the buffer
```

- No sorting required (faster)
- Requires a hash table in memory (can be large)

#### 3.8.3 Why Spark prefers HashAggregate

```text
HashAggregate:                         SortAggregate:
┌──────────────────────┐              ┌──────────────────────┐
│ No sort needed       │              │ Must sort first      │
│ O(1) lookup per row  │              │ O(n log n) sort cost │
│ ✓ Faster             │              │ ✗ Slower             │
├──────────────────────┤              ├──────────────────────┤
│ Needs hash table     │              │ No hash table needed │
│ in memory            │              │ Memory-efficient     │
└──────────────────────┘              └──────────────────────┘
```

HashAggregate is preferred because it avoids the expensive sort step. However, Spark falls back to SortAggregate in two cases:

#### 3.8.4 When Spark falls back to SortAggregate

1. **Case 1: Hash table doesn't fit in memory:** If the number of unique keys is very large, the hash table grows too big for the executor's memory.
2. **Case 2: Non-fixed-size data types in the aggregation buffer**: For HashAggregate, Spark uses **UnsafeRow** format — a compact, off-heap memory representation that bypasses JVM garbage collection (part of **Project Tungsten**). This format works by allocating fixed-size memory slots for each aggregation buffer.

```text
Fixed-size types (work with UnsafeRow):
┌──────────┬───────┬──────────────────────────────────┐
│ Type     │ Bytes │ Why it works                     │
├──────────┼───────┼──────────────────────────────────┤
│ Int      │ 4     │ Always 4 bytes, update in-place  │
│ Long     │ 8     │ Always 8 bytes, update in-place  │
│ Double   │ 8     │ Always 8 bytes, update in-place  │
│ Boolean  │ 1     │ Always 1 byte, update in-place   │
└──────────┴───────┴──────────────────────────────────┘

Variable-size types (problematic):
┌──────────┬───────────┬──────────────────────────────────────┐
│ Type     │ Bytes     │ Why it doesn't work                  │
├──────────┼───────────┼──────────────────────────────────────┤
│ String   │ varies    │ "hi" = 2 bytes, "hello world" = 11   │
│ Array    │ varies    │ Can't allocate fixed-size slot       │
│ Map      │ varies    │ Can't update in-place                │
└──────────┴───────────┴──────────────────────────────────────┘
```

If any column in the aggregation buffer has a variable-size type, Spark cannot use UnsafeRow efficiently and falls back to SortAggregate.

#### 3.8.5 Why our query uses SortAggregate

In our `main.py`, we aggregate over all fields — including String columns like `l_comment`, `l_returnflag`, `l_shipinstruct`, and `l_shipmode`:

```python
result = df.groupBy("l_linenumber").agg(
    F.sum("l_quantity"),         # Long → fixed ✓
    F.avg("l_extendedprice"),    # Double → fixed ✓
    F.sum("l_discount"),         # Double → fixed ✓
    F.min("l_returnflag"),       # String → variable ✗
    F.min("l_shipinstruct"),     # String → variable ✗
    F.min("l_comment"),          # String → variable ✗
    ...
)
```

```text
Aggregation buffer contains String types (min of l_comment, etc.)
  → UnsafeRow can't handle variable-size types efficiently
  → Spark falls back to SortAggregate

If we only aggregated numeric columns:
  F.sum("l_quantity"), F.avg("l_extendedprice"), F.sum("l_discount")
  → All fixed-size types
  → Spark would use HashAggregate ✓
```

> Be mindful of data types when writing aggregations. If you only need numeric aggregations, avoid including String columns — Spark will reward you with the faster HashAggregate.

### 3.9. CollectLimit (10)

```text
+- CollectLimit (10)

What: Take only the first 10 rows
Why:  Corresponds to .show(10) in our code
```

This limits the output to 10 rows and sends them from the executor back to the driver for display.

---

### 3.10. ResultQueryStage (11) + AdaptiveSparkPlan (18)

```text
AdaptiveSparkPlan (18)
+- ResultQueryStage (11), Statistics(sizeInBytes=8.0 EiB)

What: The outermost wrapper nodes
```

- **AdaptiveSparkPlan** — indicates this plan uses AQE (Adaptive Query Execution), meaning Spark can modify the plan at runtime.
- **ResultQueryStage** — the final AQE stage that produces the query result. The `sizeInBytes=8.0 EiB` is an estimate (often inaccurate for the outer stage — Spark uses a default placeholder when it can't estimate).
