# Cluster Architecture

## 1. Machine Cluster vs Spark Cluster — They Are NOT the Same

The **Docker Compose** file defines a **cluster of machines** (containers). This is **not** the Spark cluster itself.

The **Spark cluster** is a set of **JVM processes** (driver + executors) that run *on top of* the machine cluster. The Spark cluster manager requests resources (CPU, memory) from the machines.

```text
┌─────────────────────────────────────────────────────────────────┐
│              Machine Cluster (Docker Containers)                │
│                                                                 │
│  ┌───────────────┐   ┌───────────────┐   ┌───────────────┐      │
│  │ spark-master  │   │spark-worker-1 │   │spark-worker-2 │      │
│  │ (container)   │   │ (container)   │   │ (container)   │      │
│  │ 4 cores, 4g   │   │ 4 cores, 4g   │   │ 4 cores, 4g   │      │
│  └───────┬───────┘   └───────┬───────┘   └───────┬───────┘      │
│          │                   │                   │              │
│  ┌───────┴───────────────────┴───────────────────┴───────────┐  │
│  │             Spark Cluster (JVM Processes)                 │  │
│  │                                                           │  │
│  │  ┌──────────────────┐  ┌────────────┐  ┌────────────┐     │  │
│  │  │ Cluster Manager  │  │ Executor 1 │  │ Executor 2 │     │  │
│  │  │ (master:7077)    │  │ (JVM)      │  │ (JVM)      │     │  │
│  │  │                  │  │ 2 cores,2g │  │ 2 cores,2g │     │  │
│  │  └──────────────────┘  └────────────┘  └────────────┘     │  │
│  │         ↑                   ↑                ↑            │  │
│  │    Manages resources    Runs tasks       Runs tasks       │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**Key difference**:

| | Machine Cluster (Docker) | Spark Cluster (JVM) |
| --- | --- | --- |
| **What** | Containers providing hardware resources | JVM processes doing computation |
| **Defined by** | `docker-compose.yaml` | `spark-submit` / SparkSession config |
| **Resources** | `SPARK_WORKER_CORES=4`, `SPARK_WORKER_MEMORY=4g` | `spark.executor.cores=2`, `spark.executor.memory=2g` |
| **Manager** | Docker | Standalone / YARN / Kubernetes |

> The worker machine offers 4 cores and 4g, but your Spark app might only request 2 cores and 2g per executor. The Spark cluster manager allocates a **subset** of the machine's resources.

## 2. Docker Compose Services & Exposed Ports

```text
docker-compose.yaml
 │
 ├── spark-master (container)
 │    ├── Port 8080  →  Master Web UI (monitor workers, running apps)
 │    ├── Port 7077  →  Spark master RPC (workers & drivers connect here)
 │    ├── Volumes:
 │    │    ├── ./data → /opt/spark/work-dir/spark-warehouse/data
 │    │    └── ./spark-events → /tmp/spark-events
 │    └── Runs: start-master.sh
 │
 ├── spark-worker-1 (container)
 │    ├── No exposed ports (communicates internally via Docker network)
 │    ├── SPARK_WORKER_CORES=4, SPARK_WORKER_MEMORY=4g
 │    ├── Volumes: same as master
 │    └── Runs: start-worker.sh spark://spark-master:7077
 │
 ├── spark-worker-2 (container)
 │    ├── Same config as worker-1
 │    └── Runs: start-worker.sh spark://spark-master:7077
 │
 └── spark-history-server (container)
      ├── Port 18080 →  History Server Web UI (review past jobs)
      ├── Volumes: ./spark-events → /tmp/spark-events
      └── Runs: HistoryServer class
```

### 2.1 What are exposed ports?

**Exposed ports** (`ports: "8080:8080"`) map a port from inside the container to your host machine, so you can access it from your browser.

```text
Your browser                     Docker container
 │                                │
 │  http://localhost:8080   ──►   │  :8080  Master Web UI
 │  http://localhost:18080  ──►   │  :18080 History Server
 │                                │
 │  spark://localhost:7077  ──►   │  :7077  Master RPC (spark-submit connects here)
```

- **Port 8080** — Spark Master Web UI. See registered workers, running/completed apps, resource usage.
- **Port 7077** — Spark Master RPC. This is how `spark-submit` and workers connect to the master. Referenced in `main.py` line 20: `.master("spark://spark-master:7077")`.
- **Port 18080** — Spark History Server. Review metrics and stages of completed Spark applications.

Workers **don't expose ports** because they only need to talk to the master internally through the Docker network — you don't access them directly.

## 3. How It All Connects

```text
1. docker compose up -d
   → Starts 4 containers: master, worker-1, worker-2, history-server

2. spark-submit --master spark://spark-master:7077 main.py
   → Driver sends job to master on port 7077

3. Master allocates executors on workers
   → Worker-1 and Worker-2 each launch an executor JVM
   → Each executor gets 2 cores, 2g memory (from main.py config)

4. Executors run tasks in parallel
   → Read parquet files from mounted ./data volume
   → Perform groupBy aggregation
   → Return results to driver

5. Event logs written to ./spark-events
   → History server at :18080 reads them for post-job analysis
```

## 4. Spark History Server

The History Server is a **read-only web UI** that lets you review completed (or failed) Spark applications **after they finish**. While a job is running, you can monitor it via the Master Web UI (`:8080`). But once the job ends, that live UI disappears — the History Server preserves it.

```text
During job execution                    After job finishes
┌──────────────────────┐               ┌──────────────────────┐
│  Master Web UI       │               │  History Server      │
│  localhost:8080      │               │  localhost:18080     │
│                      │               │                      │
│  ✓ Live stages       │               │  ✓ All past jobs     │
│  ✓ Active tasks      │               │  ✓ Stage breakdown   │
│  ✓ Executor status   │               │  ✓ Task-level detail │
│                      │               │  ✓ Shuffle/spill     │
│  ⚠ Gone after job    │               │  ✓ Timeline/DAG      │
│    finishes          │               │  ✓ Always available  │
└──────────────────────┘               └──────────────────────┘
```

### 4.1 How it works

```text
Executors & Driver                  Shared Volume                History Server
 ┌──────────────┐                  ┌──────────────┐            ┌──────────────┐
 │ Running job  │  write logs ──►  │ spark-events/│  read ──►  │ Web UI       │
 │              │                  │ app-xxx.json │            │ :18080       │
 └──────────────┘                  └──────────────┘            └──────────────┘
```

1. The Spark app writes **event logs** to `./spark-events` (enabled by `spark.eventLog.enabled=true` in `main.py` line 21)
2. The History Server reads those log files from the same mounted volume
3. You browse `http://localhost:18080` to inspect stages, tasks, shuffle reads/writes, memory usage, and execution timelines

> **Why it matters**: When debugging slow jobs or data spills, the History Server is where you go to see exactly which stage/task was the bottleneck.
