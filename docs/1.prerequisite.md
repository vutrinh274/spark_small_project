# Prerequisites: Computer Architecture Basics

## 1. Quick Glossary

| Term | What it is | Analogy |
| --- | --- | --- |
| **CPU** | The brain of the computer, executes instructions | The kitchen |
| **Core** | An independent processing unit inside a CPU. More cores = more things done at once | A chef in the kitchen |
| **Process** | A running program with its own isolated memory. Can use multiple cores | A restaurant (has its own kitchen, staff, supplies) |
| **Thread** | A lightweight unit of work inside a process. Threads share the process's memory | A chef inside the restaurant вҖ” shares the same pantry with other chefs |
| **Multithreading** | Running multiple threads in one process, each on a different core | Multiple chefs cooking different dishes at the same time |
| **JVM** | Java Virtual Machine вҖ” a process that runs Java/Scala/PySpark code. A Spark executor is a JVM | The restaurant is specifically a "Java restaurant" |
| **RAM (Memory)** | Fast, temporary storage for active data. Volatile (lost on power off) | The kitchen counter вҖ” fast to reach, limited space |
| **Disk (SSD/HDD)** | Slow, permanent storage for files. Persistent (survives power off) | The warehouse вҖ” huge but slow to fetch from |
| **OOM** | Out Of Memory вҖ” program tried to use more RAM than allocated, crashes | Counter is full, dishes falling on the floor |
| **Spill to disk** | When RAM is full, Spark writes overflow data to disk (slower but avoids crash) | Counter full вҶ’ put extra ingredients on the floor (works but slow) |

## 2. CPU (Central Processing Unit)

The CPU is the "brain" of the computer. It executes instructions вҖ” math, logic, moving data around.

### 2.1. Cores

A modern CPU has multiple **cores**. Each core is an independent processing unit that can execute instructions on its own.

### 3.5. Multithreading

**Multithreading** means running multiple threads within a single process, each on a different core.

```text
CPU with 4 cores

в”Ӯ   вҶ‘    в”Ӯ   вҶ‘    в”Ӯ   вҶ‘    в”Ӯ   вҶ‘    в”Ӯ
в”ӮThread 1в”ӮThread 2в”ӮThread 3в”ӮThread 4в”Ӯ

```

This is exactly how a **Spark executor** works:

- An executor is a **JVM process**
- Each task runs on a **thread** within that process
- The number of concurrent threads = the number of cores assigned to the executor

## 4. JVM (Java Virtual Machine)

A **JVM** is a program that runs Java (and Scala, Kotlin, PySpark) code. It is itself a **process** on the machine вҖ” it gets its own memory and CPU allocation from the OS.

```text
Machine (OS)
в”ңв”Җв”Җ Process: Chrome
в”ңв”Җв”Җ Process: VS Code
в””в”Җв”Җ Process: JVM  вҶҗ This is what a Spark executor is
     в”Ӯ
     в”ңв”Җв”Җ Has its own memory (the "heap")
     в”ңв”Җв”Җ Manages its own threads
     в””в”Җв”Җ Runs your Spark task code
```

### 4.1. Why does Spark use the JVM?

Spark is written in **Scala**, which runs on the JVM. Even when you write PySpark (Python), the actual data processing happens inside a JVM process. Python just sends instructions to it.

```text
PySpark flow:

Your Python code (main.py)
        в”Ӯ
        в–ј
   Python process (driver)
        в”Ӯ  sends instructions via Py4J
        в–ј
   JVM process (driver JVM)
        в”Ӯ  distributes work
        в–ј
   JVM processes (executor JVMs on workers)
        в”Ӯ
        в–ј
   Actual data processing happens here
```

## 5. Memory (RAM) vs Disk (SSD)

### 5.1. RAM (Random Access Memory)

**RAM** is the fast, temporary storage where running programs keep their **active** data вҖ” variables, objects, data being processed right now.

- **Fast**: nanoseconds to access
- **Volatile**: data is lost when power is off
- **Expensive**: limited in size (typically 8вҖ“64 GB)

### 5.2. Disk (SSD / HDD)

**Disk** is the slow, permanent storage where files, databases, and programs are saved.

- **Slow**: milliseconds to access (1000xвҖ“100,000x slower than RAM)
- **Persistent**: data survives power off
- **Cheap**: large capacity (256 GB вҖ“ several TB)

### 6.3. Common causes of OOM in Spark

#### 6.3.1. Partition too large

A task processes one partition at a time. If a partition is bigger than the task's usable memory, it can OOM.

But why is only ~1 GB usable when the executor has 2 GB? Because of Spark's internal memory layout:

```text
Executor memory: 2 GB (spark.executor.memory = 2g)
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  Reserved: 300 MB (Spark internals)          в”Ӯ
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ  Usable: (2048 - 300) = 1748 MB              в”Ӯ
в”Ӯ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ  в”Ӯ
в”Ӯ  в”Ӯ  Unified Memory (60%): ~1049 MB        в”Ӯ  в”Ӯ
в”Ӯ  в”Ӯ  в”ңв”Җв”Җ Storage (cache/persist)           в”Ӯ  в”Ӯ
в”Ӯ  в”Ӯ  в””в”Җв”Җ Execution (shuffle/join/sort)     в”Ӯ  в”Ӯ
в”Ӯ  в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө  в”Ӯ
в”Ӯ  в”Ӯ  User Memory (40%): ~699 MB            в”Ӯ  в”Ӯ
в”Ӯ  в”Ӯ  (UDFs, internal metadata, etc.)       в”Ӯ  в”Ӯ
в”Ӯ  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ  в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

  ~1049 MB execution Г· 2 tasks = ~524 MB per task
```

#### 6.3.2. Too many cores, not enough memory

More cores per executor = more concurrent tasks = less memory per task. The executor memory stays the same, but it's split across more tasks.

```text
2 cores, 2 GB executor:                4 cores, 2 GB executor:
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ                 в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ Task 1  в”Ӯ Task 2  в”Ӯ                 в”Ӯ Task 1 в”Ӯ Task 2 в”Ӯ Task 3 в”Ӯ Task 4 в”Ӯ
в”Ӯ ~524 MB в”Ӯ ~524 MB в”Ӯ                 в”Ӯ ~262MB в”Ӯ ~262MB в”Ӯ ~262MB в”Ӯ ~262MB в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ                 в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
  вң“ Enough per task                     вҡ  Each task gets very little

Example: reading 256 MB partitions
  With 2 cores вҶ’ 524 MB per task вҶ’ fits вң“
  With 4 cores вҶ’ 262 MB per task вҶ’ tight, may OOM on shuffle/join
```

Fix: fewer cores per executor, or more memory. A common rule of thumb is ~1-2 GB of executor memory per core.
