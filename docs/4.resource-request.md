# Resource Request: How Spark Asks for Resources

## 1. The Big Picture

The **machine cluster** (Docker) provides resources. The **Spark app** requests a subset of those resources via `SparkSession` config. The **cluster manager** (standalone master) brokers between them.

```text
What the machines OFFER              What the Spark app REQUESTS
(docker-compose.yaml)                (main.py SparkSession config)

Worker 1: 4 cores, 4g memory        spark.executor.instances = 2
Worker 2: 4 cores, 4g memory        spark.executor.cores     = 2
─────────────────────────           spark.executor.memory    = 2g
Total: 8 cores, 8g memory           spark.cores.max          = 4
                                     ─────────────────────────
                                     Requests: 4 cores, 4g memory

            Spark app uses only HALF of the available resources
```

---

## 2. The Code

```python
# --- Spark cluster resource configuration ---
SPARK_EXECUTOR_INSTANCES = 2  # Number of executor JVMs to launch -  the number of executors that will be spawned for this application.
SPARK_EXECUTOR_CORES = 2  # CPU cores per executor - indicates the number of CPU cores for an executor - determines the application's parallelism capacity
SPARK_EXECUTOR_MEMORY = "2g"  # RAM per executor - indicates the amount of JVM heap memory for an executor
SPARK_CORES_MAX = (
    SPARK_EXECUTOR_INSTANCES * SPARK_EXECUTOR_CORES
)  # Total cores across cluster
SPARK_SQL_FILES_MAXPARTITIONBYTES = (
    256 * 1024 * 1024
)  # Max size of each partition when reading files (256MB)

# --- Build SparkSession with cluster and event-logging settings ---
# Event log configs enable the Spark History Server to review past jobs
spark = (
    SparkSession.builder.appName("my_app")
    .master("spark://spark-master:7077")
    .config("spark.eventLog.enabled", "true") # enabling event log, simply used for logging.
    .config("spark.eventLog.dir", "/tmp/spark-events") # configure the event log dir
    .config("spark.eventLog.logBlockUpdates.enabled", "true")
    .config("spark.executor.instances", SPARK_EXECUTOR_INSTANCES)
    .config("spark.executor.cores", SPARK_EXECUTOR_CORES)
    .config("spark.cores.max", SPARK_CORES_MAX)
    .config("spark.executor.memory", SPARK_EXECUTOR_MEMORY)
    .config("spark.sql.files.maxPartitionBytes", SPARK_SQL_FILES_MAXPARTITIONBYTES)
    .getOrCreate()
)
```

## 3. Config-by-Config Breakdown

### 3.1 `.master("spark://spark-master:7077")`

Tells the driver where to find the cluster manager. We submit the app inside the `spark-master` container, so `spark-master` resolves via Docker's internal DNS to the master container's hostname (defined in `docker-compose.yaml` line 5).

```text
spark-submit (inside spark-master container)
      │
      │  connects to spark://spark-master:7077
      ▼
Cluster Manager (master process)
      │
      │  allocates executors on workers
      ▼
Worker 1, Worker 2
```

---

### 3.2 `spark.executor.instances = 2`

How many executor JVM processes to launch across the cluster.

```text
Worker 1 (4 cores, 4g)              Worker 2 (4 cores, 4g)
┌────────────────────────┐          ┌────────────────────────┐
│                        │          │                        │
│  ┌──────────────────┐  │          │  ┌──────────────────┐  │
│  │  Executor 1      │  │          │  │  Executor 2      │  │
│  │  (JVM process)   │  │          │  │  (JVM process)   │  │
│  └──────────────────┘  │          │  └──────────────────┘  │
│                        │          │                        │
│  Remaining resources   │          │  Remaining resources   │
│  still available       │          │  still available       │
└────────────────────────┘          └────────────────────────┘
```

Each worker gets 1 executor. The master decides where to place them.

---

### 3.3 `spark.executor.cores = 2`

How many CPU cores each executor gets. This directly controls **parallelism per executor**.

The number of concurrent tasks per executor = `executor.cores ÷ task.cpus`.

```text
Executor 1 (2 cores)                Executor 2 (2 cores)
┌───────────┬───────────┐          ┌───────────┬───────────┐
│  Core 0   │  Core 1   │          │  Core 0   │  Core 1   │
│  Thread 1 │  Thread 2 │          │  Thread 3 │  Thread 4 │
│  Task A   │  Task B   │          │  Task C   │  Task D   │
└───────────┴───────────┘          └───────────┴───────────┘
    2 tasks in parallel                2 tasks in parallel

              Total: 4 tasks running simultaneously
```

> Remember: each worker machine offers 4 cores (`SPARK_WORKER_CORES=4` in docker-compose), but each executor only takes 2. The other 2 cores on each worker sit idle (available for other apps).

#### 3.3.1 `spark.executor.cores` vs `spark.task.cpus`

These are two different levels of core allocation:

- **`spark.executor.cores`** — cores assigned to the **executor**. Determines how many tasks can run in parallel.
- **`spark.task.cpus`** — cores consumed by a **single task**. Default is `1`. Almost never changed.

```text
executor.cores = 4, task.cpus = 1 (default)
┌──────────┬──────────┬──────────┬──────────┐
│  Core 0  │  Core 1  │  Core 2  │  Core 3  │
│  Task A  │  Task B  │  Task C  │  Task D  │
└──────────┴──────────┴──────────┴──────────┘
  4 ÷ 1 = 4 tasks in parallel

executor.cores = 4, task.cpus = 2
┌─────────────────────┬─────────────────────┐
│  Core 0 + Core 1    │  Core 2 + Core 3    │
│      Task A         │      Task B         │
└─────────────────────┴─────────────────────┘
  4 ÷ 2 = 2 tasks in parallel (each task uses 2 cores)
```

> `task.cpus=2` is rare — only useful when a single task internally does multithreaded work (e.g., a multithreaded ML library). We use the default of `1`, so it's not set in the code.

---

### 3.4 `spark.executor.memory = "2g"`

How much JVM heap memory each executor gets. This is the RAM budget for all tasks running in that executor.

```text
Worker machine RAM: 4g

┌──────────────────────────────────────────┐
│  Executor JVM (2g heap)                  │
│  ┌────────────────────────────────────┐  │
│  │  Reserved: 300 MB                  │  │
│  ├────────────────────────────────────┤  │
│  │  Usable: (2048 - 300) × 0.6       │  │
│  │        = 1048.8 MB                 │  │
│  │  ┌────────────────┬───────────────┐│  │
│  │  │ Storage Memory │ Execution Mem ││  │
│  │  │ (cache/persist)│ (shuffle/join)││  │
│  │  └────────────────┴───────────────┘│  │
│  ├────────────────────────────────────┤  │
│  │  User Memory: (2048-300) × 0.4    │  │
│  │             = 699.2 MB             │  │
│  └────────────────────────────────────┘  │
├──────────────────────────────────────────┤
│  Remaining worker RAM: ~2g (unused)      │
└──────────────────────────────────────────┘
```

With 2 cores per executor, each task effectively shares:

```text
~1048.8 MB usable ÷ 2 tasks = ~524 MB per task
```

---

### 3.5 `spark.cores.max = 4`

The **total** number of cores the application can use across the entire cluster.

```text
Without spark.cores.max (GREEDY — default standalone behavior):

Worker 1: 4 cores ──► Spark app takes ALL 4
Worker 2: 4 cores ──► Spark app takes ALL 4
                      Total: 8 cores grabbed!
                      ⚠ No room for other apps

With spark.cores.max = 4 (CONTROLLED):

Worker 1: 4 cores ──► Spark app takes 2 (executor 1)
Worker 2: 4 cores ──► Spark app takes 2 (executor 2)
                      Total: 4 cores used
                      ✓ 4 cores remain for other apps
```

> **Why this matters**: In standalone mode, Spark is greedy by default — it grabs every core it can find. Setting `spark.cores.max = executor_instances × executor_cores` prevents it from over-allocating. This is set to `2 × 2 = 4` in our case.

---

### 3.6 `spark.sql.files.maxPartitionBytes = 256 MB`

Controls the maximum size of each partition when reading files (like parquet). This determines how many tasks are created.

```text
Example: Reading 1 GB of parquet files

With maxPartitionBytes = 256 MB (our config):
┌────────┬────────┬────────┬────────┐
│ Part 0 │ Part 1 │ Part 2 │ Part 3 │   4 partitions = 4 tasks
│ 256 MB │ 256 MB │ 256 MB │ 256 MB │
└────────┴────────┴────────┴────────┘

With maxPartitionBytes = 128 MB (Spark default):
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ P0   │ P1   │ P2   │ P3   │ P4   │ P5   │ P6   │ P7   │  8 partitions = 8 tasks
│128MB │128MB │128MB │128MB │128MB │128MB │128MB │128MB │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

With 4 cores total and 256 MB partitions → 4 tasks → all run in 1 wave.
With 4 cores total and 128 MB partitions → 8 tasks → requires 2 waves.

> Larger partitions = fewer tasks but more memory per task. We use 256 MB here, which is double the Spark default (128 MB).

### 3.7 Event Log Configs

These three configs are purely for **monitoring** — they don't affect performance or resource allocation.

```text
spark.eventLog.enabled = true
  → Write event logs so the History Server can read them

spark.eventLog.dir = /tmp/spark-events
  → Where to write logs (mounted volume shared with History Server container)

spark.eventLog.logBlockUpdates.enabled = true
  → Also log storage block updates (cache/persist operations)
```

```text
Executor / Driver                History Server (:18080)
      │                                │
      │  writes to /tmp/spark-events   │
      │  ──────────────────────────►   │  reads and displays
      │                                │  stages, tasks, timelines
      │  (shared Docker volume)        │
```

### 3.8 Driver Resource

We leave the **driver** resource as default (not configured in code). The driver runs inside the `spark-master` container when we do `spark-submit`. It uses default memory (1g) and 1 core.

```text
spark-master container
┌──────────────────────────────────────────┐
│  Master process (cluster manager)        │
│  Driver process (our app's main())       │  ← default 1g, 1 core
│                                          │
│  The driver:                             │
│  ├── Parses your Python code             │
│  ├── Creates the execution plan          │
│  ├── Sends tasks to executors            │
│  └── Collects results (e.g., .show())    │
└──────────────────────────────────────────┘
```

---

### 3.9 Dynamic Resource Allocation (Disabled)

We don't use **dynamic allocation** (`spark.dynamicAllocation.enabled` is off by default). This means executor count is fixed at 2 for the entire job.

```text
Static allocation (our setup):         Dynamic allocation:
┌─────────────────────────┐            ┌─────────────────────────┐
│ 2 executors from start  │            │ Start with 1 executor   │
│ 2 executors during job  │            │ Scale up to 4 if busy   │
│ 2 executors at end      │            │ Scale down to 1 if idle │
│                         │            │                         │
│ ✓ Predictable           │            │ ✓ Efficient             │
│ ✓ Easy to monitor       │            │ ⚠ Harder to debug       │
│ ⚠ May waste resources   │            │ ⚠ Scaling lag           │
└─────────────────────────┘            └─────────────────────────┘
```

> We use static allocation here to make it easier to observe and understand resource usage.

## 4. Full Resource Flow Summary

```text
1. docker compose up
   → 2 workers register with master (4 cores, 4g each)

2. spark-submit main.py
   → Driver starts on spark-master container
   → Driver connects to master at spark://spark-master:7077

3. Driver requests resources:
   → 2 executors × 2 cores × 2g memory = 4 cores, 4g total
   → spark.cores.max=4 caps the request

4. Master allocates:
   Worker 1: gives 2 cores + 2g → Executor 1 (JVM)
   Worker 2: gives 2 cores + 2g → Executor 2 (JVM)

5. Data reading:
   → 20 GB of parquet, maxPartitionBytes=256MB
   → ~80 partitions created
   → 4 tasks run at a time (4 cores), ~20 waves

6. Each task:
   → 1 core, 1 thread, 1 partition (~256 MB)
   → ~524 MB usable memory per task
   → If partition > available memory → spill to disk
```
